## 关键功能差异

### 1) 路由与信息架构差异（P0）
- 基线存在桌面侧边栏信息架构：`Chat / Workspace / Resources / Settings`，且 `Workspace/Resources` 有占位页可达。
- 当前分支将 `workspace/resources` 改为强制重定向到 `/chat`，信息架构被折叠，用户从路由层面失去两个页面入口。
- 该差异会直接影响“功能入口一致性”，属于产品行为变化，而非纯视觉变化。

### 2) Chat 内容语义差异（P0）
- 基线是通用会话页（标题 Conversation，空态不预置长段文本）。
- 当前分支在空态注入了固定中文长文案作为“布局基准内容”，并替换输入占位符文案。
- 这会改变聊天首屏语义和产品文案策略，且与“仅视觉对齐”目标不一致。

### 3) 菜单行为差异（P1）
- 当前分支新增了两个浮层菜单（主菜单、区段菜单），但多数条目为静态按钮，未绑定真实业务行为。
- 代码上仅“信息”项有跳转；其它项会形成“可见但不可用”的交互预期落差。

### 4) 样式标尺差异（P1）
- 当前分支大幅采用超大字号（多处 `clamp` 上限很高），并全局替换为浅灰主题。
- 从代码参数看，样式属于“重设计”而不是“细节微调”，风险是与截图局部仍不一致时，后续难以收敛。

### 5) 测试目标迁移（P2）
- 测试由“侧边栏/Workspace 可达”迁移为“移动壳层元素存在”。
- 优点是覆盖了新布局；缺点是原有入口行为回归检查被移除，难以及时发现入口丢失问题。

## 建议修复优先级

### P0（必须先处理）
1. 明确是否允许删除 `Workspace/Resources` 入口：
   - 若不允许，应恢复可达页面（即使是移动端，也应保持入口可到达）。
   - 若允许删除，需要在需求中显式确认“功能删减”而非“视觉对齐”。
2. 移除 Chat 空态中的硬编码演示长文案，改为中性空态/历史消息渲染，避免语义偏移。

### P1（建议本轮处理）
1. 对新增浮层菜单条目补齐最小可用行为（导航或明确禁用态），避免“假入口”。
2. 把字号/间距从“超大 clamp”改为与截图实际像素更接近的固定值区间，按组件逐项校准。

### P2（后续增强）
1. 在测试中恢复“入口可达性”断言（至少覆盖 `settings`、`workspace/resources` 的预期策略）。
2. 增加视觉回归基线（如 Playwright 截图对比）防止二次漂移。

## 结论
本次变更已经完成“移动端风格化重构”，但从代码线索看，存在“视觉对齐任务”与“功能行为改动”混合的问题。建议按 P0/P1 顺序先收敛功能行为，再做像素级样式对齐。
